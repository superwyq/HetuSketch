# ADR-003: 剧情画布技术选型

## 状态

已接受（Accepted）

## 背景

剧情画布需要在 HetuSketch 现有 Electron + React + TypeScript 桌面架构中提供章节级可视化剧情编排能力，包括剧情卡拖拽、缩放、平移、连线、素材绑定、状态快照、AI 叙事生成和逻辑校验。项目当前原则是本地优先、JSON / Markdown 作为事实源、SQLite 作为可重建索引、AI 可选增强，渲染端不得直接访问文件系统或明文 API Key。

本决策覆盖前端 UI / 画布渲染、状态管理、本地持久化、SQLite 索引、AI 适配和事件处理方案。

## 考虑的选项

| 领域 | 方案 | 优点 | 缺点 |
| --- | --- | --- | --- |
| UI 组件 | Ant Design | 已在项目依赖中；表单、抽屉、标签页、按钮、通知和弹窗能力成熟；适合属性面板和工具栏 | 画布交互、自由布局和连线需要自研 |
| UI 组件 | 全量自研 CSS 组件 | 视觉和交互完全可控 | 开发成本高，易与现有工作台风格割裂 |
| 样式 | Ant Design + 自研 CSS 画布样式 | 复用通用控件，自研画布容器、卡片、锚点、选区、连线视觉 | 需要维护 Ant Design token 与自研 CSS 的一致性 |
| 画布渲染 | React DOM 卡片 + SVG 连线 | 与 React 状态天然契合；卡片表单和悬浮操作易实现；SVG 适合几十到数百条连线 | 大规模节点性能弱于专业图编辑器，需要节流与局部渲染 |
| 画布渲染 | HTML Canvas 全量绘制 | 性能上限高，适合大量图元 | 表单、选中态、无障碍、文本编辑和 Ant Design 集成成本高 |
| 画布渲染 | React Flow 类库 | 内置节点、边、缩放、拖拽、框选等能力 | 当前未引入依赖；需要评估包体、样式侵入、节点定制和长期维护成本 |
| 状态管理 | Zustand | 已在项目中使用；适合维护画布编辑态、选择态、撤销/重做和保存状态 | 复杂图编辑状态需要明确 store 切片边界 |
| 状态管理 | 组件局部状态 | 实现简单 | 多面板协作、跨组件选中态和保存状态难维护 |
| 持久化 | 本地 JSON / Markdown 事实源 | 符合项目现有原则；可读、可备份、可导出；便于 schemaVersion 演进 | 需要维护写入原子性、快照和索引同步 |
| 持久化 | SQLite 作为主存储 | 查询方便，事务能力强 | 与项目“文件为事实源”原则冲突，降低用户数据透明度 |
| 索引 | better-sqlite3 派生索引 | 已在项目中使用；适合剧情卡、时间线、状态快照、线索使用检索 | 需要处理文件变更后的重建和增量同步 |
| AI 适配 | 复用 AiService + 新增剧情上下文编译层 | 复用现有 Provider、配置、隐私和降级策略；可保持 AI 可选 | 需要定义结构化输入/输出 schema，并处理解析失败 |
| 事件处理 | Pointer Events + 键盘快捷键 + 轻量拖拽状态机 | 不新增依赖；同时覆盖鼠标、触控板和触控；便于 Electron 环境调优 | 需要自研吸附、框选、连线锚点和事件冒泡策略 |

## 决策

选择以下方案作为剧情画布首版技术路线：

1. **UI 与样式**：继续使用 Ant Design 承载工具栏、素材库、属性面板、表单、确认弹窗和通知；中央画布、剧情卡、选区、锚点、状态标记和连线样式使用自研 CSS。
2. **画布实现**：首版不引入 React Flow；采用 React DOM 渲染剧情卡、SVG 渲染连线、CSS transform 实现缩放和平移。HTML Canvas 暂不作为主渲染方案，仅保留未来用于导出图片或性能热点优化的可能。
3. **状态管理**：使用 Zustand 管理画布编辑态、选中态、视口、素材库筛选、撤销/重做栈和保存状态；事实数据仍由主进程服务读写本地文件。
4. **数据存储**：剧情画布 JSON、状态快照 JSON、章节 Markdown 和正文版本快照保存在本地文件中，SQLite / FTS5 仅保存剧情卡、时间线、状态快照和线索使用等派生索引。
5. **AI 与校验**：复用现有 AiService、validation 和 RAG 能力，在主进程新增剧情上下文编译与结果解析边界；AI 只返回 Markdown 草稿与 State Diff 建议，状态变更必须由用户确认。
6. **事件处理**：使用 Pointer Events、键盘快捷键和内部事件状态机处理拖拽、缩放、平移、框选、连线创建、素材拖入和 Ctrl+S 保存，不在首版引入专用 DnD / 图编辑库。

核心理由：

1. 最大化复用现有 Electron、React、TypeScript、Ant Design、Zustand、better-sqlite3 和本地文件架构，降低新模块引入成本。
2. 画布规模目标为 50 张卡流畅、200 张卡基础可编辑，React DOM + SVG 足以支撑首版，并能更好集成属性编辑和业务状态。
3. 保持本地优先与 AI 可选增强，避免将剧情事实、状态快照或 API Key 绑定到不可读或不可控的外部系统。

## 技术优势

- **一致性**：遵守现有三层架构，渲染端通过 `window.hetuSketch` 调用白名单 IPC，不直接访问文件系统、SQLite 或明文密钥。
- **可维护性**：卡片、连线、视口、状态快照和索引同步均可用 TypeScript 类型统一约束。
- **可扩展性**：JSON 文件包含 `schemaVersion`，未知字段读写保留；SQLite 索引可删除重建；未来可按性能瓶颈局部引入 React Flow 或 Canvas。
- **用户可控**：AI 不直接写事实源，State Diff 必须经用户确认；未配置 AI 时仍可完成画布编排、状态编辑和基础校验。
- **检索效率**：SQLite 派生索引支持按章节、角色、世界观、线索、timecode 快速定位剧情卡和状态快照。

## 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| React DOM + SVG 在大量卡片时性能下降 | 200 张卡以上拖拽、连线重绘卡顿 | 视口变换使用 CSS transform；拖拽时节流保存；只更新活动节点；后续评估虚拟化、分层 SVG 或 React Flow / Canvas |
| 自研画布交互复杂 | 框选、连线、拖拽、素材拖入事件冲突 | 建立明确事件状态机；区分画布层、卡片层、锚点层；关键交互补充用户场景测试 |
| 不引入 React Flow 可能重复造轮子 | 后续高级模板、自动布局、导出需求增加 | 首版保留适配层边界；若 P2 自动布局或大规模编辑成为核心需求，再以 ADR 评估引入 React Flow |
| JSON 事实源与 SQLite 索引不一致 | 文件保存成功但索引同步失败 | 写文件后触发索引同步；索引失败不破坏事实源；提供重建索引入口 |
| AI 输出结构不稳定 | State Diff 解析失败或正文越界发挥 | 使用结构化 schema、输出校验和降级策略；解析失败时只保留正文草稿或错误提示，不写入快照 |
| 状态字段过度自由 | 用户创建大量状态字段导致面板难用 | 状态字段按 ownerType、作用域、章节使用情况过滤展示；高级字段默认折叠 |
| 文件中泄露敏感信息 | API Key 被写入画布、快照或导出包 | AI 配置继续由主进程加密存储；画布文件只保存引用 ID、生成设置和非敏感上下文 |

## 后果

- 正面影响：剧情画布能以较低依赖成本融入现有工作台、存储、AI 和校验体系，文档与后续实现路径清晰。
- 负面影响：首版需要自研部分画布交互和性能优化，无法直接获得 React Flow 的成熟图编辑能力。
- 需要后续跟进的事项：在 Task 2 后补充共享类型、IPC 契约和 SQLite schema 文档；在 Task 3 之后基于真实性能数据评估是否需要引入专业图编辑库。
